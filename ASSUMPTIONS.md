# Assumptions

## Data Model
- Horses contain a unique id and a name.
- The id is generated by the backend when a horse is created.
- The frontend does not assign IDs for new horses.

## API Behaviour
The backend API is assumed to provide standard REST endpoints:
- GET /horse – retrieve horses
- POST /horse – create a horse
- PUT /horse/{id} – update a horse

API responses are assumed to return valid horse objects.

## Pagination
- When creating a new horse, the id is not provided by the client and is expected to be generated by the backend.
- Newly created horses are added to the client-side state after a successful save.
- The API is assumed to return the created/updated horse object.

## Validation
- Minimal validation is implemented.
- It is assumed that horse names will be provided and are valid.
- Advanced validation (duplicate names, formatting rules, etc.) is out of scope.

## Error Handling
- Basic error handling is implemented.
- Network failures, API outages, and server errors are not extensively handled due to time constraints.

## Performance
- The number of horses is assumed to remain small.
- No server-side pagination or optimization has been implemented.

## Styling
- Styling is intentionally minimal and focuses on usability rather than visual polish.
- Responsive design for mobile or tablet screen sizes was not implemented due to time constraints.
- The application is primarily intended for desktop use in its current state.
